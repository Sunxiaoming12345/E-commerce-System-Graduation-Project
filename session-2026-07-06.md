# 会话记录 — 2026-07-06

## 一、修复：购物车重复加入 500 错误

### 问题
- 第一次添加商品到购物车成功，第二次重复添加同一商品报 500
- 购物车列表始终显示为空

### 根因
1. **ShardingSphere SQL 解析器丢弃 `ON DUPLICATE KEY UPDATE` 子句**
   - Mapper XML 中定义了 `INSERT INTO cart ... ON DUPLICATE KEY UPDATE quantity = quantity + VALUES(quantity)`
   - ShardingSphere 解析后 `onDuplicateKeyColumns=Optional.empty`，实际执行的是纯 `INSERT`
   - 第二次插入同一 user_id + product_id 时触发唯一键冲突 `Duplicate entry '1-23' for key 'cart.uk_user_product'`

2. **读写分离导致购物车列表为空**
   - `write-ds` 指向 `bysj-mysql:3306`，INSERT 写入成功
   - `read-ds` 指向 `bysj-mysql-2:3307`，该容器是**独立的 MySQL 实例**，cart 表为空
   - `getCartList` 走 read-ds，始终返回空列表

### 修改的文件

#### `cart-service/src/main/java/com/example/cartservice/mapper/ShopCartMapper.java`
- 删除 `addCart()` 方法
- 新增 `insertCart()` — 纯 INSERT（不依赖 ON DUPLICATE KEY UPDATE）
- 新增 `incrementQuantity()` — 原子 UPDATE 累加数量

#### `cart-service/src/main/resources/mapper/ShopCartMapper.xml`
- 删除 `addCart` 的 `<insert>`（含 ON DUPLICATE KEY UPDATE）
- 新增 `insertCart` — 纯 INSERT
- 新增 `incrementQuantity` — `UPDATE cart SET quantity = quantity + #{quantity}`

#### `cart-service/src/main/java/com/example/cartservice/service/impl/ShopCartServiceImpl.java`
- `addCart()` 方法改为 try-INSERT + catch DuplicateKeyException + UPDATE 模式
- 不再依赖被 ShardingSphere 破坏的 ON DUPLICATE KEY UPDATE

### 部署方式
- 本地 `mvn package -DskipTests -pl cart-service -am` 编译
- JAR 传到 VM `/tmp/cart-service.jar`
- 以缓存的基础镜像 `bysj-cart-service:latest` 为 base，覆盖新 JAR 构建镜像
- `docker save → k3s ctr image import → kubectl rollout restart`

## 二、MySQL 主从复制配置

### 问题
- `bysj-mysql`（3306）和 `bysj-mysql-2`（3307）是两个独立 MySQL 实例
- 3307 不是 3306 的从库，数据完全不同步

### 操作步骤
1. 在 Master（bysj-mysql）创建复制用户 `repl@'%'`
2. 锁定表并获取 binlog 位点：`binlog.000040, pos 3495`
3. `mysqldump --all-databases --master-data=2` 导出
4. 导入到 Slave（bysj-mysql-2）
5. 使用 MySQL 8.0 语法 `CHANGE REPLICATION SOURCE TO` 配置复制
6. `FLUSH LOGS` 切到新 binlog 绕过位点边界问题
7. `START REPLICA` 启动复制

### 验证结果
- `Replica_IO_Running: Yes`
- `Replica_SQL_Running: Yes`
- `Seconds_Behind_Source: 0`
- 主从数据一致（cart 均为 6 条）

### 已知遗留
- **读写一致性**：写主库后立刻查从库可能有短暂延迟。大厂方案参考：写后短时间读主（Redis 标记）、Canal+缓存同步、半同步复制、单元化。当前毕设环境未处理此问题。
